## CLI 401 → re-auth (mine)

Implements [BDR 0029](/bdr/0029-token-invalidation-reauthentication.md) Scenarios 3, 4
under [ADR 0042](/adr/0042-detect-401-and-guide-reauthentication.md). Reuses the
`HTTP_UNAUTHORIZED` constant and the i18n message from issue 0042.

### Problem

`fetch_open_tasks` (`client.rs:128`) returns `Ok(vec![])` on any non-200, and
`collect_mine_rows` (`commands.rs:818`) calls it with `.unwrap_or_default()`. So a
revoked token makes `ac mine` print an empty task table as if the user simply had no
tasks — no signal, no recovery.

### Decision (from ADR)

A 401 from `fetch_open_tasks` is raised as a typed **`Unauthorized`** error (carried by
the existing `anyhow::Result`), not collapsed to empty. Other non-200 stays the empty
default (a normal "no tasks"/transient case). `mine_core`/`collect_mine_rows` downcast
the error and, on `Unauthorized`, print the shared re-auth message + exit non-zero. The
TUI caller keeps `.unwrap_or_default()` (swallows to empty) — **non-breaking, no TUI
change**.

### Scope

Included:

- `src/client.rs` — a public `Unauthorized` marker error type; `fetch_open_tasks`
  returns `Err(Unauthorized.into())` when `status == HTTP_UNAUTHORIZED`, keeping
  `Ok(vec![])` for other non-200.
- `src/commands.rs` — `collect_mine_rows`/`mine_core`: stop unconditionally
  `.unwrap_or_default()`-ing; inspect the `Result`, and on a downcast to `Unauthorized`
  surface the re-auth message + non-zero exit. Preserve the multi-instance aggregation
  for the success path (a 401 on the resolved instance is the reportable case for the
  single-instance `mine`; do not crash other instances' rows).
- Tests: `tests/unit/client.rs` (401 → `Err(Unauthorized)` via downcast; 404/500 →
  `Ok(vec![])`), `tests/unit/commands.rs` (`mine_core` 401 → message + non-zero exit).

Excluded: the TUI (issue 0044, where `.unwrap_or_default()` deliberately still swallows);
detail/comment (issue 0042).

### Acceptance

- AC1 — `fetch_open_tasks` typed 401 (`verify_by: test`): a MockServer 401 makes
  `fetch_open_tasks` return `Err` that downcasts to `Unauthorized`; a 404/500 returns
  `Ok(vec![])` (unchanged).
- AC2 — `mine_core` 401 (`verify_by: test`): with a revoked token, `mine_core` prints
  the shared re-auth message and exits non-zero; it does NOT print an empty table as if
  there were no tasks.
- AC3 — non-breaking TUI seam (`verify_by: test`): a caller using `.unwrap_or_default()`
  on `fetch_open_tasks` still yields an empty Vec on 401 (the error is swallowed), proving
  the change does not regress the TUI mine load.
- CC — clean code (named `Unauthorized` type; no magic `401`; no banners/commented-out)
  (`verify_by: inspection`).
- CX — complexity budget (cyclomatic ≤ 10 / ≤ 8 new; cognitive ≤ gate) (`verify_by: command`).
- TE — tests assert observable behavior (the downcast outcome; printed message; exit
  code) and survive the mutation floor (swapping 401 for another status, or collapsing
  Unauthorized back to empty, fails) (`verify_by: command`).

### Plan

1. `client.rs`: define `pub struct Unauthorized` (an `std::error::Error`/`Display` impl,
   or a thin `thiserror`-free manual impl matching the crate's style); `fetch_open_tasks`
   raises it on `HTTP_UNAUTHORIZED`, keeps `Ok(vec![])` for other non-200.
2. `commands.rs`: `collect_mine_rows`/`mine_core` inspect the `Result`; downcast to
   `Unauthorized` → re-auth message (reuse the issue-0042 key) + non-zero exit.
3. Tests: client 401-downcast + 404 regression; mine_core 401 surface; `.unwrap_or_default()` swallow.

Observable end-to-end: `ac mine` with a revoked token prints the re-auth message and
exits non-zero instead of an empty table.
